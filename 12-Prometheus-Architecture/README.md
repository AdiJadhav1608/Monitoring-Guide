# 🚀 Prometheus Architecture

---

## 📌 Introduction

Understanding Prometheus architecture is very important to know:

👉 **How data flows from systems to dashboards and alerts**

Prometheus follows a **simple yet powerful pull-based architecture**

---

## 🎯 High-Level Architecture

```text
Targets → Exporters → Prometheus Server → Storage → Grafana / Alertmanager
```

👉 This shows the complete monitoring flow 🔍

---

## 🧩 Core Components of Prometheus

---

### 🖥️ 1️⃣ Prometheus Server

- The **heart of the system ❤️**  
- Responsible for:
  - Scraping metrics  
  - Storing data  
  - Running queries  

👉 It collects data from targets at regular intervals

---

### 📡 2️⃣ Targets

- Systems or applications being monitored  
- Example:
  - Servers 🖥️  
  - Applications 🌐  
  - Databases 📦  

👉 Targets expose metrics via HTTP endpoints

---

### 🔄 3️⃣ Exporters

- Convert system data into **Prometheus-readable format**

#### 📊 Examples:
- Node Exporter → System metrics  
- Blackbox Exporter → Network checks  

👉 Exporters act as a **bridge between system and Prometheus**

---

### 📦 4️⃣ Time-Series Database (TSDB)

- Stores metrics with timestamps ⏰  

#### 🧠 Example:
```text
CPU Usage:
10:00 → 40%
10:05 → 60%
```

👉 Data is stored locally in Prometheus

---

### 🔍 5️⃣ PromQL (Query Engine)

- Used to query and analyze metrics  

#### 🧠 Example:
```text
rate(http_requests_total[1m])
```

👉 Helps in creating dashboards and alerts

---

### 🚨 6️⃣ Alertmanager

- Handles alerts generated by Prometheus  

#### 📊 Functions:
- Deduplication  
- Grouping  
- Notification (Email, Slack)

👉 Ensures proper alert delivery

---

### 📊 7️⃣ Visualization (Grafana)

- Used to create dashboards  

👉 Displays:
- Graphs 📈  
- Charts 📊  
- Real-time data  

---

## 🔄 Data Flow in Prometheus

---

### Step-by-Step Flow:

1️⃣ Targets expose metrics (via `/metrics`)  
2️⃣ Prometheus scrapes metrics periodically  
3️⃣ Data is stored in TSDB  
4️⃣ PromQL queries analyze data  
5️⃣ Grafana visualizes metrics  
6️⃣ Alerts sent via Alertmanager  

---

## ⚙️ Example: Scrape Configuration

```yaml
scrape_configs:
  - job_name: 'node_exporter'
    static_configs:
      - targets: ['localhost:9100']
```

### 📖 Explanation:
- `job_name` → Identifies the target group  
- `targets` → Systems to monitor  
- Prometheus pulls data from `/metrics` endpoint  

---

## 🚨 Key Architecture Concepts

---

### 🔄 Pull Model

- Prometheus pulls metrics  
- More reliable than push  

---

### ⏱️ Scrape Interval

- Frequency of data collection  
- Example: every 15 seconds  

---

### 🏷️ Labels

- Add metadata to metrics  
- Help in filtering and querying  

---

## ⚠️ Limitations

- ❌ Local storage only (by default)  
- ❌ Needs integration for long-term storage  
- ❌ Not designed for logs  

---

## 🧪 Interview Questions

### ❓ What are the main components of Prometheus?

Prometheus Server, Targets, Exporters, TSDB, PromQL, Alertmanager, and Grafana.

---

### ❓ What is the role of exporters?

Exporters convert system data into a format that Prometheus can understand.

---

### ❓ How does Prometheus collect data?

Using a pull-based model by scraping metrics from targets.

---

## 🚀 Summary

- Prometheus uses pull-based architecture 🔄  
- Server collects and stores metrics 📦  
- Exporters expose system data 📡  
- PromQL used for querying 🔍  
- Grafana for visualization 📊  
- Alertmanager for alerts 🚨  

👉 **Simple architecture, powerful monitoring system**
