# рҹҡҖ Prometheus Architecture

---

## рҹ“Ң Introduction

Understanding Prometheus architecture is very important to know:

рҹ‘ү **How data flows from systems to dashboards and alerts**

Prometheus follows a **simple yet powerful pull-based architecture**

---

## рҹҺҜ High-Level Architecture

```text
Targets вҶ’ Exporters вҶ’ Prometheus Server вҶ’ Storage вҶ’ Grafana / Alertmanager
```

рҹ‘ү This shows the complete monitoring flow рҹ”Қ

---

## рҹ§© Core Components of Prometheus

---

### рҹ–ҘпёҸ 1пёҸвғЈ Prometheus Server

- The **heart of the system вқӨпёҸ**  
- Responsible for:
  - Scraping metrics  
  - Storing data  
  - Running queries  

рҹ‘ү It collects data from targets at regular intervals

---

### рҹ“Ў 2пёҸвғЈ Targets

- Systems or applications being monitored  
- Example:
  - Servers рҹ–ҘпёҸ  
  - Applications рҹҢҗ  
  - Databases рҹ“Ұ  

рҹ‘ү Targets expose metrics via HTTP endpoints

---

### рҹ”„ 3пёҸвғЈ Exporters

- Convert system data into **Prometheus-readable format**

#### рҹ“Ҡ Examples:
- Node Exporter вҶ’ System metrics  
- Blackbox Exporter вҶ’ Network checks  

рҹ‘ү Exporters act as a **bridge between system and Prometheus**

---

### рҹ“Ұ 4пёҸвғЈ Time-Series Database (TSDB)

- Stores metrics with timestamps вҸ°  

#### рҹ§  Example:
```text
CPU Usage:
10:00 вҶ’ 40%
10:05 вҶ’ 60%
```

рҹ‘ү Data is stored locally in Prometheus

---

### рҹ”Қ 5пёҸвғЈ PromQL (Query Engine)

- Used to query and analyze metrics  

#### рҹ§  Example:
```text
rate(http_requests_total[1m])
```

рҹ‘ү Helps in creating dashboards and alerts

---

### рҹҡЁ 6пёҸвғЈ Alertmanager

- Handles alerts generated by Prometheus  

#### рҹ“Ҡ Functions:
- Deduplication  
- Grouping  
- Notification (Email, Slack)

рҹ‘ү Ensures proper alert delivery

---

### рҹ“Ҡ 7пёҸвғЈ Visualization (Grafana)

- Used to create dashboards  

рҹ‘ү Displays:
- Graphs рҹ“Ҳ  
- Charts рҹ“Ҡ  
- Real-time data  

---

## рҹ”„ Data Flow in Prometheus

---

### Step-by-Step Flow:

1пёҸвғЈ Targets expose metrics (via `/metrics`)  
2пёҸвғЈ Prometheus scrapes metrics periodically  
3пёҸвғЈ Data is stored in TSDB  
4пёҸвғЈ PromQL queries analyze data  
5пёҸвғЈ Grafana visualizes metrics  
6пёҸвғЈ Alerts sent via Alertmanager

## вҡҷпёҸ Example: Scrape Configuration

```yaml
scrape_configs:
  - job_name: 'node_exporter'
    static_configs:
      - targets: ['localhost:9100']
```

### рҹ“– Explanation:
- `job_name` вҶ’ Identifies the target group  
- `targets` вҶ’ Systems to monitor  
- Prometheus pulls data from `/metrics` endpoint

## рҹҡЁ Key Architecture Concepts

---

### рҹ”„ Pull Model

- Prometheus pulls metrics  
- More reliable than push  

---

### вҸұпёҸ Scrape Interval

- Frequency of data collection  
- Example: every 15 seconds  

---

### рҹҸ·пёҸ Labels

- Add metadata to metrics  
- Help in filtering and querying  

---

## вҡ пёҸ Limitations

- вқҢ Local storage only (by default)  
- вқҢ Needs integration for long-term storage  
- вқҢ Not designed for logs

## рҹ§Ә Interview Questions

### вқ“ What are the main components of Prometheus?

Prometheus Server, Targets, Exporters, TSDB, PromQL, Alertmanager, and Grafana.

---

### вқ“ What is the role of exporters?

Exporters convert system data into a format that Prometheus can understand.

---

### вқ“ How does Prometheus collect data?

Using a pull-based model by scraping metrics from targets.

---

## рҹҡҖ Summary

- Prometheus uses pull-based architecture рҹ”„  
- Server collects and stores metrics рҹ“Ұ  
- Exporters expose system data рҹ“Ў  
- PromQL used for querying рҹ”Қ  
- Grafana for visualization рҹ“Ҡ  
- Alertmanager for alerts рҹҡЁ  

рҹ‘ү **Simple architecture, powerful monitoring system**
